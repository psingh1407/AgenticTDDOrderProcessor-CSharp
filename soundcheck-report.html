✅ Mutation Score 85% — Above 80% Target

Stryker ran 105 effective mutants: 82 killed, 23 survived. The score is 7 points below the recommended threshold. The root cause is EmptyOrder_HasTotalOfZero — a zero-kill test that asserts order.Total == 0 for an empty product list. No mutation can break this because summing an empty collection always returns 0. The 23 surviving mutants are likely concentrated in ShippingCalculator and DiscountCalculator boundary conditions (off-by-one in thresholds, sign flips in rate arithmetic).

Fix

1. Delete the zero-kill test — its scenario is already covered by OrderWithOneProduct_CalculatesTotalAfterDiscount.

2. Re-run Stryker, inspect the survivors report, add one targeted assertion per surviving mutant.

P3

ℹ️ 11 Dead-Code Flags — Mostly False Positives

Roslynator flagged 11 symbols as unused. All are required but accessed via JSON serialisation or ASP.NET minimal-API model binding — neither of which Roslynator's static analysis can see. Confirmed false positives: Product.Color/Name/Size/Material (JSON body), DiscountContext constructor (bound by minimal API), Order.FinalTotal and DiscountResult.FinalTotal (serialised to response), Destination.Domestic (enum binding). Only worth suppressing if the dead-code gate is automated in CI.

Fix

Suppress with [UsedImplicitly] (JetBrains annotations) or a Roslynator suppression file, OR exclude the minimal-API layer from the dead-code scan.

# Optional: add suppression comment to silence per-symbol
// roslynator: disable RCS1163
P3

ℹ️ No Architecture Tests — Layer Boundaries Unguarded

Domain, persistence, and API all live in a single project. There are no NetArchTest.Rules tests to enforce that domain code cannot import persistence types, or that persistence cannot call the API layer. Currently this is fine — the design is clean. But as the project grows, accidental cross-layer coupling will be invisible without automated guards.

Later

Add an OrderProcessor.ArchTests xUnit project. Use NetArchTest to assert Domain does not depend on Persistence and Persistence does not depend on Program.
